INFORME GENERAL DE LA APLICACION SOFTMUSIC
Fecha de analisis: 2026-06-11
Proyecto: /srv/http/SoftMusic

============================================================
1. RESUMEN EJECUTIVO
============================================================

SoftMusic es una aplicacion de reproduccion musical local/offline.
El proyecto contiene dos versiones principales:

- Version movil Android nativa.
- Version desktop para Linux/PC.

La aplicacion esta pensada para usuarios que quieren escuchar musica
almacenada localmente en su dispositivo, sin depender de streaming,
cuentas externas o conexion permanente a internet.

El sistema ofrece funciones clasicas de reproductor musical:

- Biblioteca de canciones locales.
- Reproduccion, pausa, anterior y siguiente.
- Busqueda por cancion, artista, album o carpeta.
- Organizacion por artistas, albumes y carpetas.
- Playlists personalizadas.
- Favoritos.
- Temas visuales.
- Caratulas.
- Reproduccion en segundo plano.
- Controles multimedia del sistema.
- Modo DJ/crossfade.

No se detecto una landing page web implementada dentro del repositorio.
No existen archivos tipicos de una web como package.json, index.html,
vite.config, next.config, CSS, JavaScript o TypeScript.

Para una landing page, el proyecto ya tiene una propuesta de valor clara:
SoftMusic puede presentarse como un reproductor musical local, privado,
rapido, personalizable y disponible tanto para Android como para desktop.

============================================================
2. OBJETIVO DEL SISTEMA
============================================================

El objetivo principal de SoftMusic es permitir que el usuario reproduzca
su musica local de manera sencilla, visual y organizada.

El sistema evita depender de servicios externos. La musica se obtiene
equipo desktop.

El producto se puede enfocar en los siguientes mensajes:

- Tu musica local en un reproductor moderno.
- Sin streaming obligatorio.
- Sin cuentas de usuario.
- Sin depender de internet para escuchar tus canciones.
- Experiencia nativa en Android y desktop.
- Organizacion por canciones, artistas, albumes, carpetas y playlists.
- Personalizacion visual con temas.
- Integracion con controles del sistema.

============================================================
3. VERSIONES DISPONIBLES
============================================================

3.1 Version Android movil

La version movil esta desarrollada como aplicacion Android nativa.
Utiliza Kotlin, Jetpack Compose, Material 3 y Media3/ExoPlayer.

Esta version esta orientada a telefonos y dispositivos Android.
Obtiene canciones desde MediaStore, que es el sistema de Android para
acceder a archivos multimedia del dispositivo.

Caracteristicas principales de la version movil:

- Interfaz mobile-first.
- Permisos de lectura de audio.
- Biblioteca musical desde MediaStore.
- Reproduccion con ExoPlayer.
- Servicio de reproduccion en segundo plano.
- MediaSession para controles multimedia del sistema.
- Notificaciones de reproduccion.
- Favoritos.
- Playlists.
- Temas claros, oscuros y variantes visuales.
- Soporte de caratulas mediante Coil.

3.2 Version desktop

La version desktop esta desarrollada con Kotlin JVM y Compose Desktop.
Esta orientada principalmente a Linux/PC.

El usuario selecciona una carpeta local y la aplicacion escanea archivos

Caracteristicas principales de la version desktop:

- Interfaz desktop con Compose Desktop.
- Seleccion de carpeta musical local.
- Escaneo de subcarpetas y archivos de audio.
- Lectura de metadatos con Jaudiotagger.
- Reproduccion mediante VLC como backend principal.
- Reproduccion mediante MPV como backend experimental.
- Integracion MPRIS en Linux.
- Controles multimedia del entorno de escritorio.
- Preferencias guardadas localmente.
- Cache de caratulas.
- Favoritos.
- Playlists.
- Busqueda y filtros.
- Temas visuales.

6. ARQUITECTURA DEL SISTEMA
============================================================

SoftMusic tiene una arquitectura separada por plataforma:

- shared contiene modelos y reglas comunes.
- app contiene la experiencia Android.
- desktop contiene la experiencia desktop.

La version Android se aproxima a una arquitectura MVVM ligera:

- MainActivity inicia la aplicacion, permisos y preferencias.
- MusicViewModel centraliza estado de UI, reproduccion, cola, favoritos
  y playlists.
- MusicRepository obtiene canciones desde MediaStore.
- MusicService mantiene la reproduccion en segundo plano.
- SoftMusicApp define pantallas y componentes principales de UI.

La version desktop concentra gran parte de la UI y flujo en Main.kt:

- Main.kt contiene estado, navegacion, UI y comportamiento principal.
- DesktopMusicScanner escanea carpetas y extrae metadatos.
- DesktopAudioPlayer controla reproduccion con VLC.
- MpvPlaybackBackend controla reproduccion con MPV.
- DesktopMprisService integra controles multimedia en Linux.
- DesktopPreferencesStore guarda preferencias locales.

El modulo shared contiene entidades y reglas reutilizables:

- Song.
- MusicPlaylist.
- MusicFolder.
- Reglas de cola y reproduccion.

7. FUNCIONALIDADES PRINCIPALES
============================================================

7.1 Biblioteca musical local

La aplicacion permite acceder a canciones almacenadas localmente.

En Android, las canciones se consultan desde MediaStore.
En desktop, las canciones se obtienen escaneando una carpeta local.

Esta funcionalidad es el centro del producto y permite comunicar que
SoftMusic no depende de streaming para funcionar.

7.2 Reproduccion musical

El usuario puede reproducir, pausar, avanzar, retroceder y controlar el
progreso de la cancion.

En Android se usa Media3/ExoPlayer.
En desktop se usa VLC o MPV como backend externo.

7.3 Reproduccion en segundo plano

La version Android incluye un MusicService para mantener la reproduccion
cuando la app no esta en primer plano.

Tambien integra MediaSession para que el sistema operativo pueda mostrar
controles de reproduccion.

7.4 Controles del sistema

Android puede mostrar controles en notificaciones o pantalla bloqueada.

Desktop puede integrarse con MPRIS en Linux, permitiendo controles desde
el entorno de escritorio.

7.5 Busqueda

El usuario puede encontrar canciones usando criterios como:

- Titulo.
- Artista.
- Album.
- Carpeta.

7.6 Filtros y vistas de organizacion

La musica puede visualizarse o agruparse por:

- Canciones.
- Artistas.
- Albumes.
- Carpetas.
- Favoritos.
- Playlists.

7.7 Favoritos

El usuario puede marcar canciones como favoritas para acceder rapidamente
a ellas.

7.8 Playlists

El usuario puede crear listas de reproduccion y organizar canciones.

Las playlists se guardan localmente.

7.9 Temas visuales

La aplicacion tiene opciones de apariencia, incluyendo temas claros,
oscuros y variantes visuales.

Esto ayuda a que la app tenga una experiencia personalizable y atractiva.

7.10 Caratulas

La aplicacion maneja imagenes de caratulas para mejorar la experiencia
visual.

Android usa Coil para cargar imagenes.
Desktop extrae y cachea caratulas cuando estan disponibles.

7.11 Modo DJ / crossfade

SoftMusic incluye un modo DJ o crossfade para transiciones entre
canciones.

Esta funcion es atractiva para marketing, pero debe comunicarse con
cuidado si aun no esta totalmente validada en todos los escenarios.

8. FUNCIONAMIENTO EN ANDROID MOVIL
============================================================

Flujo general de uso:

1. El usuario abre SoftMusic en Android.
2. La app solicita permisos para acceder a archivos de audio.
3. Si el permiso es concedido, la app consulta canciones desde MediaStore.
4. Se muestra la biblioteca musical.
5. El usuario puede buscar, filtrar, reproducir, marcar favoritos o crear
   playlists.
6. Al reproducir, MusicService mantiene el audio activo en segundo plano.
7. El sistema Android puede mostrar controles multimedia.
8. El usuario puede cambiar temas y preferencias.

Permisos Android relevantes:

- READ_MEDIA_AUDIO en Android 13 o superior.
- READ_EXTERNAL_STORAGE en Android 12 o inferior.
- POST_NOTIFICATIONS en versiones recientes cuando aplica.

Componentes clave Android:

- MainActivity.kt
- SoftMusicApp.kt
- MusicViewModel.kt
- MusicService.kt
- MusicRepository.kt
- AndroidManifest.xml

Ventajas de la version movil:

- Experiencia nativa.
- Integracion con el sistema Android.
- Reproduccion en segundo plano.
- Interfaz pensada para pantallas tactiles.
- Acceso a biblioteca local del telefono.

Puntos a validar en movil:

- Telefonos pequenos.
- Telefonos grandes.
- Orientacion horizontal.
- Tablets.
- Permisos denegados.
- Biblioteca vacia.
- Biblioteca con muchas canciones.
- Reproduccion en segundo plano prolongada.
- Controles de notificacion y pantalla bloqueada.

13. ANALISIS RESPONSIVE Y VERSIONES
============================================================

Android movil:

La UI esta orientada a mobile-first. Usa Compose, listas verticales,
bottom sheets y componentes adecuados para interaccion tactil.

No se detecto una estrategia fuerte de breakpoints para tablet o layouts
amplios. Para una landing se puede afirmar que existe version movil, pero
conviene validar telefonos pequenos, grandes, landscape y tablet antes de
prometer soporte adaptativo avanzado.

Desktop:

La version desktop tiene interfaz propia de escritorio. Usa paneles,
sidebar y barras de control.

Algunos valores parecen fijos, como anchos de sidebar o paneles. Esto
puede limitar el comportamiento en ventanas pequenas. Conviene probar
redimensionamiento antes de usar capturas en pantallas muy estrechas.

============================================================
15. RENDIMIENTO
============================================================

Factores positivos:

- No depende de llamadas remotas para cargar musica local.
- Android usa MediaStore, que es el mecanismo nativo de consulta.
- Desktop escanea archivos locales.
- No hay backend externo que pueda bloquear el uso principal.

Riesgos de rendimiento:

- Bibliotecas muy grandes pueden afectar escaneo inicial.
- Extraccion de metadatos y caratulas puede ser costosa en desktop.
- VLC/MPV como procesos externos pueden tener variaciones por sistema.
- Listas muy grandes requieren cuidado en la UI.
- Crossfade puede aumentar complejidad de reproduccion.

Recomendaciones:

- Medir escaneo con bibliotecas grandes.
- Mostrar progreso o estados claros durante carga.
- Cachear caratulas y metadatos cuando sea posible.
- Probar consumo de memoria en desktop.
- Probar reproduccion prolongada.

17. RIESGOS TECNICOS
============================================================

Riesgo alto:

- No existe landing page web real dentro del proyecto.

Impacto:

No se puede publicar una pagina promocional sin crear primero una
estructura web o elegir una tecnologia.

Recomendacion:

Definir si la landing sera parte de este repositorio o un proyecto web
separado.

Riesgo alto:

- Desktop depende de VLC o MPV externos.

Impacto:

El usuario podria instalar la app pero no reproducir musica si no tiene
el backend necesario.

Recomendacion:

Comunicar requisitos claramente en landing y documentacion.

Riesgo medio:

- Desktop puede no adaptarse bien a ventanas muy pequenas.

Impacto:

La experiencia podria romperse o ser incomoda en resoluciones bajas.

Recomendacion:

Probar y ajustar layouts adaptativos.

Riesgo medio:

- Android no muestra una estrategia avanzada para tablets.

Impacto:

Puede funcionar, pero no necesariamente aprovechar pantallas grandes.

Recomendacion:

Validar tablets antes de usar claims de compatibilidad avanzada.

Riesgo medio:

- Main.kt desktop concentra mucha logica.

Impacto:

Puede dificultar mantenimiento y evolucion.

Recomendacion:

Separar pantallas, componentes y estado en archivos mas pequenos si se
planean cambios grandes.

Riesgo medio:

- Modo DJ/crossfade es complejo.

Impacto:

Puede tener errores en cambios de cancion, repeat, shuffle o canciones
cortas.

Recomendacion:

Probarlo antes de destacarlo como funcion estable.

Riesgo bajo:

- Faltan assets comerciales.

Impacto:

La landing necesitara capturas, mockups y textos finales.

Recomendacion:

Crear screenshots reales en Android y desktop.
